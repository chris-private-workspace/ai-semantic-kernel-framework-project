## 0. Background

The Skills System epic (Sprints 57.113-117) gave the platform CC-style skills: a system-global `SkillRegistry` loads bundled `SKILL.md` files (`Skill(name, description, instructions)`, a frozen value object ‚Äî `registry.py:53-59`); a cheap `## Available Skills` block rides the system prompt; the model lazy-loads a skill's full instructions via the `read_skill` Cat-2 tool (`tool.py:50-98`) and then **self-implements** the instructions. A per-tenant DB overlay (`tenant_skills`, 57.114) + a `/skill-name` force-load (57.115) + a user-turn chip (57.116) + count/size quotas (57.117) round it out ‚Äî but **every skill is instructions-only**: there is no executable code, no bundled script, no resources. `Skill` has no `script` field anywhere (backend / DB / frontend).

CC's skills can bundle **executable scripts** (a `.py`/`.sh` alongside `SKILL.md`) that the model can RUN as part of using the skill ‚Äî deterministic tooling shipped with the skill, vs re-deriving the same logic in prose every invocation. `AD-Skills-Bundled-Scripts` (logged 57.113, the largest remaining CC-parity gap per `claudedocs/5-status/agent-harness-cc-parity-20260607.md` row 9) asks for this. This slice closes the **system-bundled leg**.

The harness ALREADY has a sandboxed-execution primitive ‚Äî the `python_sandbox` Cat-2 tool (`tools/exec_tools.py:27-97`) runs LLM-supplied Python via a `SandboxBackend` (`tools/sandbox.py`): `DockerSandbox` (Sprint 52.5 P0 #17 ‚Äî real OS isolation: `--cap-drop=ALL`, `read_only`, `network=none`, non-root uid 65534, `pids_limit`) when a Docker daemon is reachable, else a legacy `SubprocessSandbox` (dev/CI ‚Äî best-effort, weak on Windows) via `default_sandbox()`. It is gated by `RiskyActionDetector` (Cat 9, a 9-pattern STRING deny-list, ESCALATE-not-BLOCK, per-tenant-configurable) + the Cat-2 tool permission gate. So bundled-script execution is **NOT greenfield sandbox work** ‚Äî it reuses this primitive. The only new code: (1) let a skill carry a script, (2) a tool that runs the skill's SERVER-controlled script, (3) the demo asset + tests + the drive-through.

### Design decision (a `Skill.script` field populated from a sibling `<stem>.py` in `bundled/` ‚Üí a NEW `run_skill_script(skill_name)` Cat-2 tool that runs the skill's server-controlled script via the existing `SandboxBackend`, registered via `make_default_executor`'s `skill_registry` opt-in (auto-PASS like `read_skill`), feeding stdout/stderr back ‚Üí a demo skill + script ‚Üí a real chat drive-through; NO DB / NO wire / NO frontend / NO design-time sandbox change)

- **US-1 is "a skill can carry a script"** (Cat 5): `Skill` gains `script: str | None = None` (frozen, defaulted ‚Üí existing skills + the tenant overlay's `Skill(name, description, instructions)` construction are unaffected, `script` defaults `None`). `SkillRegistry.from_dir`, after parsing `<stem>.md`, looks for a sibling `<stem>.py`; if present, reads its source into `Skill.script`. Bundled-only (the disk loader); tenant skills (`with_overlay`) carry no script (`None`). The existing 2 skills (no sibling `.py`) ‚Üí `script=None`.
- **US-2 is "the model can run it"** (Cat 2): a NEW `RUN_SKILL_SCRIPT_TOOL_SPEC` (input `{skill_name: string}` ‚Äî NO `code` arg; the script is server-controlled) + `make_run_skill_script_handler(registry, sandbox)`: looks up the skill; an unknown skill OR a skill with `script is None` returns a recoverable message (NOT an exception); otherwise runs `skill.script` via `sandbox.execute(...)` (the existing `SandboxBackend`, network-blocked) and returns a JSON `{stdout, stderr, exit_code, duration_seconds, killed_by_timeout}` (mirrors the `python_sandbox` handler output). `read_only=False`, `risk_level=MEDIUM` (code execution, like `python_sandbox`), `hitl_policy=AUTO`.
- **US-3 is "wired into the chat path"** (Cat 2 / platform): `make_default_executor`, when `skill_registry` is supplied, registers `run_skill_script` ALONGSIDE `read_skill` (`_register_all.py:295-297`), binding a `default_sandbox()` backend (Docker if reachable, else Subprocess). The chat path's registry-derived permission matrix auto-grants a PASS rule (handler.py builds rules from `registry.list()`) ‚Äî the same mechanism that lets `read_skill` (57.113) and `python_sandbox` (memory-wired) run on chat. **Day-0 critical**: confirm the registry-derived matrix PASSes a MEDIUM/exec tool, not only read-only (the `python_sandbox` precedent strongly implies yes).
- **US-4 is "a demo bundled skill + script"**: a NEW `bundled/<name>.md` (frontmatter + instructions telling the model: when asked for the canonical digest, call `run_skill_script('<name>')` and report what it returns) + `bundled/<name>.py` (a self-contained script that COMPUTES + prints a deterministic value the LLM cannot fabricate ‚Äî e.g. `hashlib.sha256(b"...").hexdigest()`). Self-contained (no input/args this slice). The output being a runtime computation (not a constant the model could recall) makes the drive-through a real execution proof.
- **US-5 is "tests"**: unit (registry ‚Äî `from_dir` reads a sibling `<stem>.py` into `script`; a skill with no sibling ‚Üí `script is None`; `with_overlay` keeps `script=None` for tenant skills) + unit (tool ‚Äî `run_skill_script` runs a skill's script via an injected fake/real sandbox ‚Üí stdout; an unknown skill ‚Üí message; a skill with `script is None` ‚Üí message) + (executor ‚Äî `make_default_executor(skill_registry=...)` registers `run_skill_script`) + integration (the tool runs the demo bundled script via the real `default_sandbox()` and returns the computed digest ‚Äî gated/skipped if the sandbox is unavailable in CI).
- **US-6 is "the drive-through"** (real chat + real sandbox): with a real chat session, ask the model to use the demo skill ‚Üí it `read_skill`s (optional) ‚Üí calls `run_skill_script('<name>')` ‚Üí the sandbox executes the bundled script ‚Üí the digest flows back ‚Üí the model reports it ‚Üí verify the reported value EQUALS the locally-computed `hashlib.sha256(...).hexdigest()` (provably script-produced). Note which backend ran (Docker vs Subprocess). Screenshots + observed-vs-intended in progress.md. The FIRST main-flow proof of sandboxed execution.
- **Rejected / deferred**: tenant-authored scripts (the `tenant_skills` table stays text-only ‚Äî running customer code is a separate, larger security decision); multi-file skill bundles / resource files (CC's fuller shape ‚Äî `Skill.script` is one script this slice); an admin authoring UI for scripts (the Skills tab stays text-only); model-supplied script INPUT/args (`run_skill_script` takes only `skill_name` ‚Äî a self-contained script this slice); running the bundled script's source through the `RiskyActionDetector` deny-list (it ESCALATEs on patterns a legitimate script may use ‚Äî the trust model is supply-chain-at-authorship + sandbox-isolation-at-runtime, deny-list is for LLM-supplied `python_sandbox` code; revisit if tenant-authored lands); any wire / codegen / SSE / migration / frontend change.

### 3.2 The `run_skill_script` tool (US-2)
