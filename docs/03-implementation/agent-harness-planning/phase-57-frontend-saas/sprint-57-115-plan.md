(f) **No new SSE wire event** (count stays 24, zero codegen) — force-load rides the existing chat POST; the list endpoint is a plain GET. (g) **`read_skill` stays available** — force-load guarantees presence; the model can still self-select other skills via `read_skill`. (h) **DRY** — a shared `render_skill_instructions(skill)` helper feeds BOTH the `read_skill` tool body and the force-load block (no duplicated formatting).

### 3.1 `render_skill_instructions` + `build_handler` force-load (US-1)

- **`render_skill_instructions(skill: Skill) -> str`** (`registry.py`): return `f"# Skill: {skill.name}\n\n{skill.instructions}"` (the body the `read_skill` tool already emits, minus the trailing "Follow these instructions…" line which stays tool-local). Pure, no I/O. `make_read_skill_handler` (`tool.py`) refactors to `f"{render_skill_instructions(skill)}\n\nFollow these instructions for the current task."` → the tool's returned string is **byte-identical** (Day-0 reads the exact current string to guarantee this).
- **`build_handler(..., force_load_skill: str | None = None)`** (`handler.py`): after the existing `system_prompt = f"{system_prompt}\n\n{render_catalog_block(skill_registry.list())}"` (when a registry is present), if `force_load_skill` and `skill_registry` and `(skill := skill_registry.get(force_load_skill))`: append `f"\n\n## Active Skill\nThe user explicitly selected the \"{skill.name}\" skill for this request. Follow its instructions:\n\n{render_skill_instructions(skill)}"`. The append is the ONLY change to the function. (`build_real_llm_handler` threads the param through if it's the real-LLM builder; Day-0 confirms which builder the router calls + whether one delegates to the other.)
- The forced skill is also still listed in the catalog block (harmless — the catalog names+describes; the `## Active Skill` block gives the full body + marks it active). `read_skill` remains registered and callable (the model may still self-select other skills).

### 3.2 `GET /api/v1/chat/skills` list endpoint (US-2)

- **`GET /skills`** on the chat `router` (`api/v1/chat/router.py`): `db: AsyncSession = Depends(get_db_session)` + `current_tenant` (the chat POST's tenant dependency — Day-0 reads the exact name); `registry = await resolve_tenant_skill_registry(db, current_tenant)`; return `ChatSkillsResponse(skills=[ChatSkillItem(name=s.name, description=s.description) for s in registry.list()])`. Read-only, no audit, no mutation. (Same per-tenant source as the chat POST → the picker always matches what a chat turn would resolve, 約束 2.)
- **Pydantic** (`schemas.py`): `class ChatSkillItem(BaseModel): name: str; description: str` · `class ChatSkillsResponse(BaseModel): skills: list[ChatSkillItem]`. NO `instructions` (bodies load on force-load / `read_skill`).
